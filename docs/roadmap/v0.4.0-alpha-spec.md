# Conchordal v0.4.0 Alpha Specification Draft

Status: Draft
Audience: Codex, Claude Code, maintainers
Scope: v0.4.0 alpha release planning

## Release Thesis

The academic concept has already been stated in the paper, and the paper render
is fixed at v0.3.0. v0.4.0 is not a paper-reproduction release.

v0.4.0 is a research-composer-facing alpha. The target user is not a general
musician who expects conventional composition abstractions. The target user is a
research composer who is willing to learn Conchordal's own concepts and use them
as a medium for making works.

The primary v0.4.0 goal is rhythm/harmony ecology: a scenario author should be
able to shape, move, bias, and inhabit consonance terrain while also composing
time as a living structure. Rhythm is no longer secondary in this release.
Metric beat, entrained beat, and flow timing must be part of the
composer-facing alpha surface before v0.4.0 is cut.

v0.4.0 is not the release for timbre genetics or spatialization. Those are
deferred to v0.5.0 or later.

## Non-Goals For v0.4.0

- reproducing the paper render or revalidating the paper experiments
- making the system understandable to general musicians without Conchordal
  context
- TimbreGenotype inheritance, mutation, or evolutionary timbre selection
- spatialization, panning, 3D landscape, or spatial auditory ecology
- full backward compatibility with pre-alpha scenario names or API quirks
- polished composer-facing beta workflows
- broad new DSP architecture work unless required to stabilize the rhythm,
  harmony, or life-model alpha API

## Main Gap

The implementation is ahead of the research-composer scripting model.

The current scripts can reproduce paper-derived behavior, but many examples still
read as research assays. v0.4.0 needs fewer, stronger, more extensible
Conchordal-native concepts:

- how to create a consonance field
- how to seed, bias, modulate, and invert consonance terrain
- how to let voices move through that terrain
- how to make voices survive, disappear, or respawn according to consonance
  viability
- how to keep rhythm hooks available without letting rhythm design dominate this
  release
- how to hear the difference between fixed placement, density placement, adaptive
  movement, and consonance-biased respawn

The API also exposes several important features that are not documented as part
of the complete Rhai reference. This must be fixed before release.

## Proposed Public Concept Layers

### Layer 1: Consonance Defaults

These are the first concepts a v0.4.0 research composer should see:

- `derive(harmonic).sustain()`
- `create(species, count).place(consonance(root))`
- `create(species, count).place(consonance_density(min, max))`
- `set_harmonic_mirror(value)`
- `consonance_movement()`
- `movement_glide(tau_sec)`

Goal: a user can create a Consonance Field, place voices by field strategy or
Consonance Density, and hear voices move through the field without learning
metabolism, LOO, or respawn internals.

`consonance(root)` and `consonance_density(min, max)` must remain distinct.
`consonance(root)` is field-based placement. `consonance_density(min, max)` is
density-based spawn placement from a normalized PMF.

`consonance_movement()` is the standard movement entrypoint. It means:

- pitch is free to move
- the default movement core is hill-climb over the Consonance Field
- movement is applied as glide, not gate-snap
- default glide time is `0.30` seconds

`movement_glide(tau_sec)` overrides the default glide time. Crowding, LOO,
global peaks, ratio candidates, move cost, and other tuning controls stay out of
the standard entrypoint and remain Layer 4 research controls or Layer 2 ecology
controls.

### Layer 2: Consonance Ecology

These express the paper-derived alpha behavior as compositional controls:

- viability: `consonance_viability`, `viability_rate`, `dissonance_cost`
- replacement: `respawn_consonance`, `respawn_random`, `respawn_capacity`
- self-awareness and crowding: `leave_self_out`, `crowding`, `crowding_target`
- routing roles: `send(habitat_bus)`, `send(presentation_bus)`

Goal: a user can make a population that changes over time because consonance
changes its viability.

`consonance_viability(low, high)` is environment-relative by default. A voice
should not be sustained merely by consonance that it contributes to itself; it
should be sustained by the consonance it has with its environment. In v0.4.0 this
uses the existing real-time approximate LOO path internally. The public API
should expose the compositional concept, not the approximation mechanism.

If an explicit override is needed, prefer a scope-level API:

- `viability_scope("environment")` for counterfactual self-excluded viability
- `viability_scope("total")` for self-included total-field viability

`selection_approx_loo` should not appear in curated demos. It may remain as an
implementation/reference control during the transition, but it is not the
composer-facing name.

### Layer 3: Rhythm Ecology

These are already present as mechanisms, but they must be redesigned into a
composer-facing rhythm ecology before v0.4.0 is released:

- pulse articulation: `repeat`, `pulse`, `gates`, `field`
- phase alignment: `rhythm_coupling_vitality`, `rhythm_reward`
- social onset coupling: `social`
- scaffold comparison controls: `set_scaffold_off`, `set_scaffold_shared`,
  `set_scaffold_scrambled`

Goal: v0.4.0 can demonstrate metric beat, entrained beat, flow timing,
and integrated rhythm/harmony ecology as musical concepts rather than only
mechanism-level controls.

### Layer 4: Research Controls

These remain available, but should not dominate the first demo path:

- implementation/reference LOO controls, including `selection_approx_loo` and
  exact scan modes
- low-level hill-climb and peak-sampler tuning
- `pitch_core`, `pitch_apply_mode`, `pitch_glide`, `global_peaks`,
  `ratio_candidates`, `move_cost`, `improvement_threshold`
- external rhythm scaffold comparisons as experimental controls:
  `set_scaffold_off`, `set_scaffold_shared`, `set_scaffold_scrambled`
- detailed energy constants
- report/telemetry knobs

Goal: advanced users can reproduce and inspect paper-derived behavior without
forcing every scenario author to think in experiment terminology.

Scaffold functions are not part of the stable v0.4.0 composer-facing rhythm
foundation. They inject an external temporal scaffold by overriding the rhythm
model and should be documented as research comparison controls for testing
coupling, phase reward, and synchronization behavior.

## Perception And Counterfactual Design

This design decision is fixed for v0.4.0 and later development.

The shared Conchordal world is grounded in the habitat. The main
generator Landscape must come from the mixed habitat bus:

```text
voices -> habitat mix -> NSGT -> H/R/C Landscape
```

This keeps the generator world audio-grounded without making hidden control
tones audible. `send(habitat_bus)` removes a voice from the presentation
bus only. `send(presentation_bus)` removes a voice from the habitat bus
and therefore from the generator Landscape.

Per-voice leave-one-out viability is different. It is a counterfactual question:
what would this voice's consonance relation to the world be if its own
contribution were absent? That should not require live per-voice solo rendering
and NSGT in the main runtime.

The long-term architecture is:

```text
shared world:
  actual mixed audio -> NSGT -> Consonance Field / Density / Rhythm

per-voice counterfactual:
  voice spectral footprint -> subtract or suppress own contribution -> viability
```

`selection_approx_loo` is the v0.4.0 real-time approximation to this
counterfactual. It uses the voice pitch and a harmonic proxy to suppress likely
self-contribution. This is good enough to fix the public meaning of
environment-relative viability, but it is not the final fidelity target.

The final fidelity target is a sound-body spectral footprint. This footprint is
not a static harmonic table. It is a time-dependent counterfactual contribution
model evaluated from the voice's current sound body, articulation state, pitch,
and timbre phenotype.

- sine bodies contribute one peak
- harmonic bodies contribute partials with current amplitudes
- modal bodies contribute modal partials, including independent decay state
- inharmonic bodies may contribute non-integer partial ratios or absolute modal
  frequencies
- future timbre genotypes contribute phenotype-derived partial/noise footprints

The footprint abstraction must allow partial-specific envelopes, independent
decay rates, time-varying amplitudes, non-integer partials, and non-tonal/noise
components. It may be rendered internally as Log2Space-aligned density or as a
small component list that is projected into Log2Space, but the public meaning is
the same: the model estimates this voice's current contribution to the
habitat-derived Landscape without solo-rendering and NSGT in the live path.

Rendered per-voice LOO, where each voice is solo-rendered and analyzed through
NSGT, belongs in offline/reference validation or `conchordal-render`, not in the
live instrument's default runtime.

Therefore, v0.4.0 should stabilize the public semantics now:

- the generator world is actual mixed habitat-bus audio
- Consonance Viability is environment-relative by default
- the live implementation may use a harmonic proxy
- later releases may improve the proxy into sound-body footprints without
  changing the composer-facing concept

## NeuralRhythms And Predictive Fields

This design note fixes how rhythm, future prediction, and consonance should
relate.

`NeuralRhythms` is not the predictive field itself. It is the temporal model
that makes predictive fields usable:

- phase says where the system is in a rhythmic cycle
- frequency gives candidate future times such as the next gate
- magnitude says whether the rhythm is present
- precision says how much to trust rhythm-based prediction
- prediction error says when to shorten or weaken the prediction horizon

A Predictive Consonance Field is a time-indexed acoustic terrain:

```text
C(f, t) = predicted consonance at frequency f and time t
```

It should be built by the GeneratorModel from the current habitat-derived
Landscape, NeuralRhythms, scheduled phonation events, and eventually predicted
voice footprints. The current implementation already has a narrow version of
this idea: `GeneratorModel` estimates the next gate from `NeuralRhythms.theta` and
predicts a `pred_c_field_level_scan` for that time. v0.4.0 does not need to
complete this architecture, but the public concepts should not block it.

Keep these axes orthogonal:

```text
source:
  field now          actual habitat-bus audio observed through NSGT
  predictive future  estimated field at a future event time

scope:
  total              includes the voice's own contribution
  environment        self-excluded counterfactual

target:
  movement           where the voice should move
  timing             when the voice should sound
  viability          whether the voice is supported by the environment
```

This implies the future API direction:

```rhai
.consonance_movement()
.movement_horizon("next_gate")

.consonance_viability(0.30, 0.80)
.viability_scope("environment")
.viability_horizon("now")
```

The horizon axis must not be conflated with the self-exclusion axis.
`viability_scope("environment")` answers "with or without my own contribution?"
`viability_horizon("now" | "next_gate" | ...)` answers "which time should be
evaluated?"

Rhythm/harmony fusion should be understood as a policy-level composition:

```text
rhythm   = when a voice contributes to the world
harmony  = which spectral components it contributes
fusion   = choosing when and with what spectrum the voice enters the field
```

For movement, future prediction is often musically natural: if the next field
event happens at the next gate, movement may choose a pitch using the predicted
field at that gate rather than only the current field. For viability, the
default should remain current habitat-derived environment-relative
viability unless a future horizon is explicitly requested. This keeps survival
grounded in the actual generator world while leaving room for expected-future
viability as an advanced composition tool.

Prediction must avoid self-fulfilling loops. The runtime should use staged
snapshots:

1. observe the current habitat-derived Landscape from actual mixed habitat-bus audio
2. update NeuralRhythms from the field rhythm stream
3. build a short-horizon predictive field from observed state, rhythm phase, and
   known scheduled or footprint-derived events
4. let voices decide movement, timing, and viability against that snapshot
5. render the next audio hop, then observe again

When rhythmic confidence is high, prediction may look toward the next gate or a
short number of cycles. When confidence is low or prediction error is high, the
horizon should collapse toward the current field.

## API Work Needed

### Required

1. Audit Rhai reference against registered functions.
   - `docs/rhai_book/src/reference/life.md` must include every public v0.4.0
     method that samples use.
   - Current reference coverage includes sample-facing lifecycle, viability,
     rhythm, respawn, scaffold, and control-update APIs.
   - Keep this audit alive as new v0.4.0 public functions are promoted.

2. Define the best long-term API without preserving backward compatibility by
   default.
   - Follow the scenario-authoring principle: keep simple things simple, and
     complex things possible.
   - Prefer a small number of extensible primitives over many narrow aliases.
   - Keep implementation names only when they are also good research-composer
     concepts.
   - During alpha, breaking renames are allowed when they improve the long-term
     scripting surface.

3. Implement and document the accepted v0.4.0 public names.
   - `consonance_density_pmf` -> `consonance_density`
   - `survival_signal` -> `consonance_viability`
   - `continuous_recharge_rate` -> `viability_rate`
   - default `consonance_viability` to environment-relative viability using the
     real-time approximate LOO path
   - add `viability_scope("environment")` and `viability_scope("total")` only if
     explicit scope override is needed
   - `respawn_peak_bias` -> `respawn_consonance`
   - `set_harmonicity_mirror_weight` -> `set_harmonic_mirror`
   - add `consonance_movement()` as the standard movement entrypoint, equivalent
     to `pitch_mode("free")`, hill-climb movement, glide application, and
     `pitch_glide(0.30)`
   - add `movement_glide(tau_sec)` as the public glide control
   - rename `sync` (currently registered for both `SpeciesHandle` and
     `GroupHandle` in `src/life/scripting/engine.rs`). `sync` is a Rhai
     reserved keyword, so the LSP definition file (`rhai-defs/conchordal.d.rhai`)
     cannot expose it and editor completion misses it. Suggested rename:
     `phase_sync` or `lock_in`. No current sample or test uses it.

4. Keep the Rhai LSP definition file in sync with `register_fn` calls.
   - `rhai-defs/conchordal.d.rhai` is auto-generated by
     `cargo run --bin gen_rhai_defs` and committed.
   - `Rhai.toml` at the repo root wires this into the rhai-lsp workspace.
   - When adding a public function in `src/life/scripting/`, regenerate the
     definition file in the same commit. CI drift check guards against
     forgetting.
   - Avoid registering functions whose names collide with Rhai reserved
     keywords (full list at https://rhai.rs/book/language/keywords.html).

5. Clarify draft-only versus live-patchable behavior.
   - Rhythm, lifecycle, respawn, and brain choices are draft-only today.
   - Pitch, brightness, density/crowding, and several movement knobs are live
     patchable.
   - This distinction must be visible in both reference docs and examples.

### Optional If Time Allows

1. Add higher-level helper aliases only when they remove real friction.
   Do not add aliases just to hide the system. v0.4.0 should remain honest
   about the ALife model.

2. Add one or two convenience presets for common alpha behaviors, if they
   replace repeated sample boilerplate.

## Rhythm Redesign Blocker

The previous two-demo curated path is paused. `pulse_foundation.rhai` is useful
evidence that timing controls exist, but it is not sufficient for the v0.4.0
release bar.

The rhythm redesign plan is tracked in
`docs/roadmap/v0.4.0-rhythm-redesign.md`. It is now a release blocker.

Required rhythm families:

1. Metric beat: a Western-music-like pulse that is immediately legible.
2. Entrained beat: pulse or synchronization emerging from agent coupling,
   vitality, reward, and social timing.
3. Flow timing: non-metric but structured onset patterns like water,
   fireflies, rain, or wind-driven clusters.

The release needs two different demo roles: a functional integration demo that
shows the APIs working together, and a musical showcase that proves the result
can stand as a listening experience.

## Demo Work Needed

The release needs a small curated set of demos that are better than the current
research assay scripts.

### Required Demos

1. Metric beat demo
   - Shows a Western-music-like beat that is easy to parse.
   - Uses stable attack and duration roles without hiding Conchordal concepts.
   - Candidate path: `samples/04_ecosystems/metric_beat_foundation.rhai`.

2. Entrained beat demo
   - Shows pulse or synchronization emerging from agent coupling and survival.
   - Uses rhythm coupling, rhythm reward, social timing, metabolism, and
     optionally respawn.
   - Candidate path: `samples/04_ecosystems/entrained_beat.rhai`.

3. Flow timing demo
   - Shows rain/river-like non-metric timing patterns that are structured but
     not grid-locked or slowly tempo-warped.
   - Candidate path: `samples/04_ecosystems/flow_timing_field.rhai`.

4. Functional rhythm/harmony integration demo
   - Shows that the v0.4.0 APIs compose in one stable ecology.
   - Demonstrates metric beat, entrained beat, flow timing, consonance movement,
     and harmonic field shifts without requiring a finished musical form.
   - Candidate path: `samples/04_ecosystems/rhythm_harmony_ecology.rhai`.

5. Musical ecology showcase
   - The flagship v0.4.0 alpha listening demo.
   - Shows a musically legible Conchordal result, using only the features that
     serve the form rather than trying to demonstrate every API at once.
   - Should still make consonance, rhythm, and life-process ecology audible.
   - Candidate path: `samples/04_ecosystems/conchordal_ecology.rhai`, or a
     dedicated etude if the flagship needs to be separated from feature
     coverage.

No required curated v0.4.0 demo should exceed 120 seconds.

## Release Acceptance Criteria

v0.4.0 can be called alpha when all of the following are true:

1. The redesigned rhythm/harmony curated demo set exists and passes
   compile-only checks.
2. The functional integration demo demonstrates metric beat, entrained beat,
   flow timing, consonance movement, and harmonic-field response together.
3. A separate musical showcase demonstrates consonance control, rhythm
   emergence, and ecology in a musically legible way.
4. The rhythm API supports metric beat, entrained beat, and flow timing
   as composer-facing concepts.
5. The Rhai Quick Start explains the core v0.4.0 concepts without relying on
   paper experiment names.
6. The Rhai API reference includes every public method used by curated samples.
7. Existing research comparison samples still compile.
8. `cargo fmt --all` passes.
9. `cargo clippy -- -D warnings` passes.
10. `RUST_BACKTRACE=1 cargo test -- --nocapture` passes before any source-code
   release commit.
11. The README and web roadmap no longer describe an outdated pre-alpha state.
