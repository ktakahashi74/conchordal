# Conchordal v0.4.0 Rhythm Architecture Audit

Status: M1 draft
Updated: 2026-05-08

## Summary

The current codebase already contains substantial rhythm machinery. The release
problem is not the absence of timing logic. The problem is that the public
surface collapses rhythm into `pulse`, `gates`, `sync`, `social`, and low-level
Kuramoto/metabolism controls. Those names do not yet describe the musical
concepts that v0.4.0 must demonstrate.

Key findings:

- `TimingField` in `src/life/phonation_engine.rs` is the best existing center
  for a composer-facing timing abstraction.
- `PhonationConfig` is richer than the public `PhonationSpec`; composite
  clocks, subdivisions, internal phase, and sub-theta modulation already exist
  internally but are not cleanly exposed through Rhai.
- `KuramotoCore` already provides endogenous phase, coupling, vitality, reward,
  and phase noise. This should become the basis of entrained beat rather than
  remaining a set of mechanism-level knobs.
- `SocialDensityTrace` gives local onset-density feedback, but it is currently
  a per-hop support trace rather than a persistent rhythm field.
- Reports already write raw onset samples and Kuramoto order into runtime data,
  but they do not summarize rhythm as beat stability, synchronization
  emergence, burstiness, or 1/f slope.
- Flow timing is the weakest area. Existing pink noise affects timbre or
  phase fluctuation, not onset/rate structure as a musical timing concept.

## Public API And Internal Model

| Current surface | Current implementation | Composer-facing role | M1 decision |
|---|---|---|---|
| `repeat`, `pulse(rate)`, `gates(n)` | `WhenSpec::Pulse` becomes a time-based accumulator with a theta-gate clock; `gates` maps to fixed gate durations. | Low-level metric articulation primitive. | Keep, but do not make it the main v0.4 rhythm story by itself. |
| `sync(depth)` | Adds cosine sub-theta weighting to the pulse path, but the name collides with Rhai syntax. | Legacy accent/phase-weighting primitive for metric beat. | Keep registered for now; expose `accent(depth)` for new scripts. |
| `social(coupling)` | Uses recent onset density as an onset-weight multiplier. | Seed of entrainment feedback. | Keep as low-level; M2 should decide whether it becomes part of a named entrained beat preset. |
| `field()` duration plus `field_window`, `field_curve`, `field_drop` | Field-dependent duration rule driven by timing-field excitation and slope. | Timing-field articulation, but the name collides conceptually with Consonance Field. | Keep internally; M2 should rename or wrap it if it remains public. |
| `rhythm_freq`, `rhythm_sensitivity`, `k_omega`, `base_sigma`, `gate_thresholds` | Articulation/Kuramoto parameters in `ArticulationCoreConfig::Entrain`. | Mechanism controls for endogenous rhythm. | Keep available for advanced scripts; do not make them the entry API. |
| `rhythm_coupling`, `rhythm_coupling_vitality`, `rhythm_reward` | Coupling/reward controls for phase, vitality, and attack-phase matching. | Basis of entrained beat. | Promote through a higher-level entrainment vocabulary. |
| `set_scaffold_off`, `set_scaffold_shared`, `set_scaffold_scrambled` | External scaffold comparison modes. | Research controls for comparison conditions. | Keep research-only; not part of the composer-facing v0.4 foundation. |
| `TimbreGenotype::jitter` | 1/f pink-noise FM strength. | Timbral/pitch irregularity. | Do not reuse as the primary flow-timing feature. |

## Source Inventory

### Scenario And Rhai API

Primary files:

- `src/life/scenario.rs`
- `src/life/control_adapters.rs`
- `src/life/scripting/mod.rs`
- `src/life/scripting/engine.rs`

Current public shape:

- `WhenSpec` exposes `Once` and `Pulse { rate, sync, social }`.
- `DurationSpec` exposes `WhileAlive`, `Gates`, and `Field`.
- `PhonationSpec` contains only `when` and `duration`.
- `PhonationConfig` contains richer internal timing concepts:
  `OnsetConfig`, `PhonationClockConfig`, `SubThetaModConfig`,
  `DurationConfig`, `SocialConfig`, and `PhonationMode`.

Adapter finding:

- `phonation_config_from_spec` currently maps public `WhenSpec::Pulse` to
  `OnsetConfig::Accumulator`, `PhonationClockConfig::ThetaGate`,
  optional `SubThetaModConfig::Cosine`, and `SocialConfig`.
- The public `PhonationSpec` path does not expose
  `PhonationClockConfig::Composite`, subdivisions, or internal phase clocks.

Decision:

- Treat the existing public API as the low-level compatibility layer during the
  alpha phase.
- Build the v0.4 composer-facing API as a cleaner layer over the richer
  internal `PhonationConfig`, not as more flags on `pulse`.

### Phonation Engine

Primary file:

- `src/life/phonation_engine.rs`

Current capabilities:

- `TimingField` combines theta gate openness, envelope openness, optional
  social density, and extra predicted-gate gain into an excitation value.
- `ThetaGateClock` provides gate-aligned timing.
- `PhonationClock::Composite` already supports a theta gate clock plus
  subdivision and internal phase clocks.
- `SubThetaMod::CosineHarmonic` supports within-gate weighting.
- `OnsetRule::Accumulator` gives deterministic time-based onset accumulation
  with gate-weighting and refractory gates.
- `DurationRule::Field` can extend or shorten durations based on timing-field
  excitation and slope.

Gaps:

- Flow timing needs a timing-level stochastic or colored-noise onset/rate
  rule. It should not be faked with timbre jitter.
- The field-duration behavior can be musically useful, but the name `field`
  is too ambiguous now that Consonance Field is the central harmonic concept.
- Composite timing is underused because it is not reachable from the simple
  Rhai surface.

Decision:

- Promote `TimingField` as the architectural center for rhythm control.
- Use existing accumulator and composite-clock mechanics for metric beat before
  adding new machinery.
- Add flow timing at the onset/rate layer, with seeded deterministic tests.

### Articulation And Entrained Beat

Primary file:

- `src/life/articulation_core.rs`

Current capabilities:

- `KuramotoCore` tracks rhythm phase/frequency and updates phase with temporal
  coupling, environment gates, phase noise, vitality, and rhythm reward.
- `rhythm_coupling_vitality` makes coupling depend on vitality.
- `rhythm_reward(..., "attack_phase_match")` connects articulation timing to
  metabolism.
- The code already exposes Kuramoto order at population/runtime level.

Gaps:

- The current Rhai names are implementation terms. They do not tell a composer
  how to create an emergent beat.
- The entrainment story spans articulation, metabolism, social density, and
  reports, but there is no named preset or model boundary tying them together.

Decision:

- Use `KuramotoCore` as the basis of entrained beat.
- Add a higher-level Rhai concept in M2 that sets the current parameters as a
  coherent ecology, while keeping advanced controls available.

### Social Timing

Primary files:

- `src/life/population/social.rs`
- `src/life/social_density.rs`

Current capabilities:

- `build_social_trace_from_batches` builds onset-density traces from recent
  phonation batches.
- `SocialDensityTrace` bins and normalizes onset density.
- Smoothing exists in `SocialDensityTrace::from_onsets`.

Gaps:

- `social_trace_params` currently uses no smoothing.
- Social timing is rebuilt as support data for a hop; it is not yet a named,
  inspectable rhythm field.
- There is no report that tells whether social timing created synchronization
  or only increased onset count.

Decision:

- Keep the current social trace as a low-risk substrate.
- M2 should decide whether v0.4 needs a persistent `RhythmField`, or whether a
  named entrainment preset over the existing trace is enough for alpha.

### Reporting

Primary files:

- `src/life/report.rs`
- `src/runtime/mod.rs`

Current capabilities:

- `JsonlReporter` writes scene markers, spawn, respawn, death, onset, and
  group-step records.
- `OnsetSample` includes time, group, voice, generation, frequency, strength,
  PLV, scaffold mode, and scaffold phase.
- Runtime frames already carry `kuramoto_order_r` and
  `phonation_onsets_in_hop`.
- `summarize_groups` reports group-level pitch/consonance state.

Gaps:

- No named rhythm summary report.
- No inter-onset interval distribution.
- No beat stability score.
- No synchronization emergence time.
- No burstiness score for flow timing.
- No approximate 1/f slope estimate for onset-rate variation.

Decision:

- M3 should add rhythm summaries before final demos are judged by audition.
- The first report implementation can be derived from existing onset records;
  it does not need to change audio-thread behavior.

## Required Rhythm Families

### Metric Beat

Existing foundation:

- time-based `OnsetRule::Accumulator`
- theta-gate clock
- fixed gate durations
- sub-theta cosine weighting
- internal composite clocks

Missing for v0.4:

- a clear composer-facing way to express beat divisions, accents, and optional
  offbeats
- beat-stability and IOI reporting
- a demo that sounds deliberately articulated rather than merely pulsed

Recommended M2 direction:

- expose a small metric-beat vocabulary that maps to existing clocks and
  accumulator logic
- keep `pulse` and `gates` as primitive operations

### Entrained Beat

Existing foundation:

- Kuramoto phase and coupling
- vitality-modulated coupling
- rhythm reward through attack-phase matching
- social onset-density feedback
- Kuramoto order telemetry

Missing for v0.4:

- a named ecology preset or builder
- synchronization emergence reporting
- a demo where population-level beat appears without relying on an external
  scaffold

Recommended M2 direction:

- wrap the existing Kuramoto/social/metabolism controls in a coherent
  entrainment concept
- keep scaffold modes only as comparison controls

### Flow Timing

Existing foundation:

- seeded randomness in voice/genotype paths
- pink-noise timbral jitter
- phase noise in Kuramoto dynamics

Missing for v0.4:

- onset/rate-level colored-noise timing
- deterministic seed tests
- burstiness and approximate slope reporting
- a demo where the pattern is neither metronomic nor white-random

Recommended M2 direction:

- add a timing-level flow driver in the phonation path
- keep it separate from timbre jitter and from scaffold controls

## Test And Report Gap List

Required before M4 demos are considered release candidates:

- unit tests for any public metric-beat wrapper mapping into
  `PhonationConfig`
- deterministic seeded tests for flow timing
- report tests for IOI summaries and beat stability
- report tests for synchronization emergence on a controlled entrained-beat
  sample
- report tests for burstiness and approximate 1/f slope on seeded flow timing
- compile-only checks for all new curated samples
- headless sample runs that emit rhythm reports for all three families

## M2 Design Questions

M2 should answer these before code changes:

1. Should the public names be family-level builders such as `metric_beat`,
   `entrained_beat`, and `flow_timing`, or smaller compositional verbs?
2. Should `field()` duration remain public, be renamed, or become an internal
   detail behind timing-field articulation?
3. Should entrained beat require a persistent `RhythmField`, or can v0.4 use
   the existing social trace plus Kuramoto telemetry?
4. How directly should rhythm viability affect survival and respawn in the
   flagship demo?
5. What minimal report schema is enough to keep rhythm demos honest without
   creating a full analysis subsystem?

## Recommended Implementation Order

1. Design the public M2 rhythm vocabulary and decide which existing low-level
   names stay documented as primitives.
2. Add rhythm summary reports from existing onset records.
3. Expose metric beat using existing accumulator, composite clock, subdivision,
   and sub-theta weighting paths.
4. Add an entrained rhythm preset over Kuramoto, vitality coupling, rhythm
   reward, and social timing.
5. Add timing-level flow onset/rate variation with deterministic seeded
   tests.
6. Rebuild the three rhythm-family demos.
7. Build the integrated rhythm/harmony flagship.

## Exit Status

M1 is sufficient to proceed to M2 design. The implementation owners are clear:

- Scenario/API: `src/life/scenario.rs`, `src/life/control_adapters.rs`,
  `src/life/scripting/mod.rs`, `src/life/scripting/engine.rs`
- Event generation: `src/life/phonation_engine.rs`
- Agent-phase entrainment: `src/life/articulation_core.rs`
- Population timing: `src/life/population.rs`, `src/life/population/social.rs`,
  `src/life/social_density.rs`
- Reports: `src/life/report.rs`, `src/runtime/mod.rs`
- Demos and docs: `samples/04_ecosystems/`, `docs/rhai_book/src/`
