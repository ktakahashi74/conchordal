# Conchordal v0.4.0 Rhythm Core Redesign

Status: M2 accepted direction; implementation in progress
Updated: 2026-05-08

## Design Decision

v0.4.0 is the first release that puts rhythm in the foreground. This is the
right time to simplify the rhythm architecture instead of adding thin wrappers
over legacy names.

Use five explicit layers:

```text
NeuralRhythms
  Perceived rhythm state extracted from the field bus.

RhythmIntent
  Composer-facing timing intention: metric, entrained, flow.

AgentPhase
  Per-agent endogenous phase and synchronization state.

TimingField
  Onset viability built from NeuralRhythms, RhythmIntent, AgentPhase, social
  timing, and consonance prediction.

OnsetProcess
  The actual triggering process: regular accumulator, phase-locked process,
  flow process, or later extensions.
```

The important boundary is that `NeuralRhythms` is perceptual state, not a
general-purpose external clock. Composer intent and experimental references
must not directly overwrite it.

## Public Rhai Surface

Use these v0.4 entry points:

- `metric_beat(rate_hz)`
- `entrained_beat(rate_hz)`
- `flow_timing(mean_rate_hz)`
- `flow_timing(mean_rate_hz, depth)`

Do not use `ecological_beat` as the API name. Ecology describes the whole
Conchordal system and the flagship demo. The rhythm-specific term is
entrainment.

Do not use `natural_timing` as the API name. The implementation is a structured
flow timing process, not a direct model of nature.

## Layer Responsibilities

### NeuralRhythms

Source:

- `DorsalStream` extracts rhythm from the field bus.
- `RhythmEngine` tracks theta, delta, envelope openness, and confidence.

Responsibility:

- represent perceived rhythm
- expose phase, frequency, magnitude, precision, and prediction error
- update from audio/field input

Non-responsibility:

- do not serve as a mutable external metronome for composer APIs
- do not carry research scaffold state

### RhythmIntent

Responsibility:

- store composer-level time intent on a species
- separate metric beat, entrained beat, and flow timing from low-level
  implementation controls
- choose the appropriate onset process and timing-field weighting

Initial variants:

- `None`
- `MetricBeat { rate_hz, accent }`
- `EntrainedBeat { rate_hz, social, vitality_coupling, reward }`
- `FlowTiming { mean_rate_hz, depth }`

### AgentPhase

Current implementation:

- `KuramotoCore` in `articulation_core.rs`

Responsibility:

- hold endogenous agent phase
- synchronize through temporal coupling
- connect timing fit to vitality and reward

Public surface:

- `entrained_beat()` is the musical entry point
- `rhythm_coupling_*`, `rhythm_reward`, `rhythm_freq`, and related controls are
  advanced overrides, not the first path

### TimingField

Current implementation:

- transient gate-indexed excitation in `phonation_engine.rs`

Responsibility:

- combine perceived rhythm openness, social timing, consonance prediction, and
  intent-level timing weights
- provide onset viability per candidate gate
- drive both onset probability and adaptive duration

Boundary:

- keep the internal name `TimingField`
- describe it publicly as onset viability or timing terrain
- do not introduce a persistent `RhythmField` in v0.4 unless implementation
  evidence makes it necessary

### OnsetProcess

Current implementation:

- `OnsetRule::Accumulator`

Needed v0.4 processes:

- regular accumulator for metric beat
- entrained accumulator for population synchronization
- flow process for clustered non-metric onset timing

The `OnsetRule` implementation can remain the code-level owner for now, but
the design role should be named `OnsetProcess`.

## Public Rhythm Modalities

### Metric Beat

Purpose:

- legible beat, subdivision, and attack/duration roles
- Western-music-friendly articulation
- synchronized with perceived neural rhythm when available

Implementation direction:

- use `RhythmIntent::MetricBeat`
- use regular onset accumulation
- align candidate gates to `NeuralRhythms`
- use accent/subdivision as explicit metric controls, not overloaded `sync`

### Entrained Beat

Purpose:

- beat appears through agent phase, social timing, vitality, and reward
- synchronization is measurable, not only described

Implementation direction:

- use `RhythmIntent::EntrainedBeat`
- configure agent phase via existing Kuramoto machinery
- use social onset density as feedback into `TimingField`
- write Kuramoto order and synchronization emergence reports

### Flow Timing

Purpose:

- non-metric but structured timing
- clustered, stationary, seed-reproducible onset behavior
- suitable for streams, droplets, flashes, swarms, and flowing textures

Implementation direction:

- use `RhythmIntent::FlowTiming`
- implement a stationary timing-level onset process with local renewal jitter
  and short rain-like clusters
- do not reuse timbre jitter as the primary timing mechanism
- do not express flow as slow tempo/rate warping over an otherwise metric pulse
- do not put long-memory pink drift directly into IOI when the musical target is
  rain or river flow
- validate with burstiness and approximate slope reports

### Additional Modalities To Preserve In The Design

The three public entry points are enough for v0.4 demos, but the architecture
should leave room for:

- field-conditioned timing: consonance prediction and timing terrain influence
  onset/duration
- lifecycle rhythm: birth, death, respawn, and survival pressure create longer
  temporal structure
- experimental reference clocks: scaffold comparisons remain research controls

## Legacy Cleanup

| Current item | Decision |
|---|---|
| `pulse(rate)` | Keep as low-level primitive. `metric_beat()` becomes the recommended entry point. |
| `accent(depth)` | Preferred low-level accent/phase weighting name for new scripts. |
| `sync(depth)` | Legacy registered name; avoid in new scripts because it collides with Rhai syntax. |
| `social(coupling)` | Keep as low-level feedback control. `entrained_beat()` sets a musical default. |
| `field()` duration | Rename or wrap later; the name collides with Consonance Field. Treat as advanced timing-duration behavior. |
| `rhythm_freq` | Advanced override for AgentPhase. |
| `rhythm_coupling_vitality` | Advanced override used by `entrained_beat()`. |
| `rhythm_reward` | Advanced override used by `entrained_beat()`. |
| `base_sigma` | Mechanism-level phase noise, not flow timing. |
| `set_scaffold_*` | Research-only reference clock; isolate from composer-facing rhythm API. |

## Report Surface Required By The Model

Report in two layers:

- `rhythm_observation`: time-series rhythm state such as Kuramoto order,
  active count, onsets in hop, and current perceived rhythm values
- `rhythm_summary`: final/window summary for onset density, IOI, beat stability,
  synchronization emergence, burstiness, approximate onset-density slope, and
  approximate IOI-sequence slope

The report surface should exist before the final demos are tuned by audition.

## Implementation Order

1. Introduce `RhythmIntent` and the new public names.
2. Keep legacy primitives working, but route new APIs through the new intent
   layer.
3. Add a flow onset process at the timing layer.
4. Add rhythm observation records and complete summary metrics.
5. Rebuild the three rhythm demos and the integrated flagship.

This release may take longer. The conceptual surface published in v0.4 is more
important than preserving pre-alpha naming quirks.
