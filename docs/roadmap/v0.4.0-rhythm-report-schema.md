# Conchordal v0.4.0 Rhythm Report Schema

Status: M3 report surface and sample report runs complete; audition pending
Updated: 2026-05-08

## Purpose

Rhythm demos must pass audition, but audition alone is not enough. v0.4.0 needs
headless rhythm summaries that make three failure modes visible:

- a metric beat that is actually unstable
- an entrained beat that never synchronizes
- flow timing that is only white-random triggering

The first implementation should derive summaries from existing report data
where possible. It should avoid audio-thread changes.

## Existing Inputs

Already available:

- `onset` records from `JsonlReporter::write_onsets`
- `OnsetSample.time_sec`
- `OnsetSample.group_id`
- `OnsetSample.voice_id`
- `OnsetSample.strength`
- `OnsetSample.plv`
- scaffold mode and scaffold phase
- runtime Kuramoto order in UI/runtime frames
- `group_step` records for group pitch/consonance summaries

Still intentionally absent from the first report surface:

- explicit rhythm-family label per group
- expected beat rate per group
- flow-timing configuration per group

M3 records global rhythm observations and summaries without the family labels.
M4/M5 can add family-aware labels only if audition shows that the reports need
that distinction.

## JSONL Records

Runtime now emits `rhythm_observation` records during execution:

```json
{
  "type": "rhythm_observation",
  "time_sec": 12.0,
  "kuramoto_order_r": 0.74,
  "kuramoto_active_count": 9,
  "onsets_in_hop": 3,
  "theta_hz": 6.1,
  "delta_hz": 1.0,
  "env_open": 0.82,
  "env_level": 0.55
}
```

At report flush, runtime emits one global `rhythm_summary` record and one
additional `rhythm_summary` per group that had onsets:

```json
{
  "type": "rhythm_summary",
  "time_sec": 60.0,
  "window_start_sec": 0.0,
  "window_end_sec": 60.0,
  "group_id": null,
  "onset_count": 142,
  "onset_density_hz": 2.3667,
  "ioi_mean_sec": 0.42,
  "ioi_cv": 0.18,
  "beat_stability": 0.82,
  "mean_plv": 0.61,
  "kuramoto_order_mean": 0.74,
  "kuramoto_order_max": 0.93,
  "sync_emergence_sec": 18.5,
  "burstiness": -0.21,
  "one_over_f_slope": -0.95,
  "ioi_one_over_f_slope": -0.88
}
```

Field meanings:

| Field | Meaning |
|---|---|
| `time_sec` | Time when the summary is emitted. |
| `window_start_sec` | Analysis window start. |
| `window_end_sec` | Analysis window end. |
| `group_id` | Optional group-specific summary. `null` means whole scenario. |
| `onset_count` | Number of onsets in the window. |
| `onset_density_hz` | `onset_count / window_duration_sec`. |
| `ioi_mean_sec` | Mean inter-onset interval, sorted by onset time. |
| `ioi_cv` | IOI standard deviation divided by mean. |
| `beat_stability` | Bounded stability score derived from IOI regularity. |
| `mean_plv` | Mean available phase-locking value across onsets. |
| `kuramoto_order_mean` | Mean population Kuramoto order in the window. |
| `kuramoto_order_max` | Maximum population Kuramoto order in the window. |
| `sync_emergence_sec` | First time the order stays above the synchronization threshold, if any. |
| `burstiness` | `(std_ioi - mean_ioi) / (std_ioi + mean_ioi)`. |
| `one_over_f_slope` | Approximate log-power/log-frequency slope of binned onset counts. |
| `ioi_one_over_f_slope` | Mean per-voice approximate log-power/log-frequency slope of the IOI sequence itself. |

## M3 Metrics

The implementation in `src/life/report.rs` and `src/runtime/mod.rs` now writes
global and group-level summaries with:

- `onset_count`
- `onset_density_hz`
- `ioi_mean_sec`
- `ioi_cv`
- `beat_stability`
- `mean_plv`
- `burstiness`
- `kuramoto_order_mean`
- `kuramoto_order_max`
- `sync_emergence_sec`
- `one_over_f_slope`
- `ioi_one_over_f_slope`

The first implementation remains intentionally approximate. `one_over_f_slope`
uses binned onset counts. `ioi_one_over_f_slope` uses per-voice sorted IOI
sequences, then averages finite voice slopes. Both use simple
log-power/log-frequency regressions as guardrails, not publication-grade
spectral estimators.

## Beat Stability

For M3, define:

```text
beat_stability = clamp(1.0 - ioi_cv, 0.0, 1.0)
```

This is intentionally simple. It catches the common failure where a supposed
metric beat has highly irregular intervals. If a later report has expected beat
rate labels, the score can be tightened without changing the record shape.

## Synchronization Emergence

For M3/M4, define:

```text
sync_emergence_sec = first t where kuramoto_order_r >= 0.70 for at least 2.0 sec
```

If no such run occurs, write `null`.

## Flow Timing Guardrails

Flow timing should not be accepted only because it sounds irregular. The
summary should show:

- nonzero burstiness
- IOI variation higher than metric beat
- finite onset/IOI slope diagnostics without requiring slow drift

Do not make the first release depend on an exact `-1.0` slope. The goal is to
distinguish structured flow timing from uniform random timing without making
the whole rhythm audibly breathe at a slow tempo.

## Tests

Required unit tests:

- regular IOIs produce high beat stability
- irregular IOIs produce lower beat stability and nonzero burstiness
- empty and single-onset windows produce defined `None` or zero values
- PLV averaging ignores missing values
- Kuramoto observations populate mean, max, and synchronization emergence
- approximate 1/f slope is finite for a multiscale onset fixture
- approximate IOI 1/f slope is finite for a multiscale IOI fixture
- seeded flow timing process is deterministic

Current report-run checks:

- metric demo emits a high-stability rhythm summary
- entrained demo emits a synchronization summary
- flow demo emits non-metric IOI variation and flow-timing guardrail metrics
- integrated ecology emits global and group rhythm summaries
